# TUPA 웹사이트 Notion CMS 스키마 설계

> 승인 전 초안 — 이 문서가 승인되기 전에는 Notion에 아무것도 생성하지 않는다.
> 목표: **코딩을 모르는 교수님이 Notion 표를 고치는 것만으로 사이트가 갱신**되고,
> 어떤 실수를 해도 사이트가 깨지지 않는 구조.

## 0. 전체 구조

```
📄 TUPA CMS (부모 페이지, integration 연결)
├── 📊 멤버
├── 📊 논문
├── 📊 소식
├── 📊 연구분야
├── 📊 연구과제
├── 📊 모집공고
├── 📊 갤러리
├── 📊 사이트 설정
└── 📄 사용 가이드 (교수님용, 스크린샷 포함)
```

**공통 원칙**

| 원칙 | 내용 |
|---|---|
| 공개 체크박스 | 모든 DB에 `공개` (Checkbox). 체크된 행만 사이트에 노출. 새 행 기본값은 미체크 → 실수로 초안이 노출되지 않음 |
| 순서 | 수동 정렬이 필요한 DB에만 `순서` (Number, 오름차순). 비워두면 맨 뒤 |
| 한글 프로퍼티명 | 교수님이 보는 이름은 전부 한글. 코드에서 매핑 |
| 폴백 대원칙 | **Notion 미연결·API 장애·빈 필드 → 코드에 내장된 현재 실데이터로 폴백.** 사이트는 어떤 경우에도 빈 화면/에러를 보여주지 않는다 |
| 반영 시간 | Next.js ISR `revalidate: 3600` — 수정 후 최대 1시간 내 반영 |

## 1. 멤버 (Members)

| 프로퍼티 | 타입 | 필수 | 설명 · 폴백 |
|---|---|---|---|
| 이름(영문) | Title | ✅ | 사이트 표기 기준. 비면 행 자체를 숨김 |
| 이름(한글) | Rich text | | 비면 영문만 표시 (현재 사이트 동작과 동일) |
| 구분 | Select | ✅ | `지도교수` `연구원` `박사과정` `석사과정` `방문` `인턴` `졸업생`. 비면 행 숨김 |
| 직함(한글) | Rich text | | 예: "박사과정 · 2023 입학". 비면 구분값으로 기본 직함 자동 표시 |
| 직함(영문) | Rich text | | 비면 구분값 기본 직함 |
| 사진 | Files | | 비면 이니셜 아바타 (현재 동작) |
| 이메일 | Email | | 비면 미표시 |
| 연구 관심사 | Multi-select | | 비면 미표시 |
| 홈페이지 | URL | | 비면 미표시 |
| Google Scholar | URL | | 비면 미표시 |
| 진로 | Rich text | | 졸업생용 "Now · …" 표시. 비면 미표시 |
| 공개 | Checkbox | ✅ | |
| 순서 | Number | | 같은 구분 안에서의 표시 순서 |

> 사진 참고: 기존 51명 사진은 저장소(`public/people/`)에 이미 있음.
> Notion `사진` 필드가 **채워져 있으면 그걸 우선** 사용, 비어 있으면 저장소 사진 → 이니셜 순.

## 2. 논문 (Publications)

| 프로퍼티 | 타입 | 필수 | 설명 · 폴백 |
|---|---|---|---|
| 제목 | Title | ✅ | 비면 행 숨김 |
| 저자 | Rich text | ✅ | 표기 순서 그대로 쉼표 구분 (예: `Kaihan Zhang, Reuben Tamakloe, Inhi Kim`). 랩 구성원 이름은 프론트에서 자동 볼드 |
| 발표처 | Rich text | | 학회/저널명. 비면 미표시 |
| 연도 | Number | ✅ | 비면 행 숨김 (연도 그룹핑 필수라서) |
| 종류 | Select | | `학회` `저널` `북챕터`. 비면 `학회` |
| 링크 | URL | | 논문 원문. 비면 Google Scholar 제목 검색 링크 자동 생성 (현재 동작) |
| DOI | Rich text | | 있으면 링크보다 우선 (`doi.org/…`) |
| 연구분야 | Relation → 연구분야 | | 연구분야 카드의 "관련 논문"에 사용. 비면 무관 |
| 대표 논문 | Checkbox | | 홈/상단 강조용 |
| 공개 | Checkbox | ✅ | |

> **설계 결정 — 멤버 Relation을 쓰지 않는 이유**: 논문 저자의 절반 이상이 외부
> 공저자라 Relation로는 저자 목록을 완성할 수 없고, 교수님이 논문 하나 넣을 때마다
> 멤버를 일일이 연결하는 부담이 생김. 텍스트 한 줄이 가장 빠르고 안전.
> 정렬은 `연도` 내림차순 고정이라 순서 필드 불필요.

## 9. 프론트 아키텍처 (코드 계약)

```
페이지 컴포넌트 ──(변경 없음)──▶ lib/content.ts  (getter 시그니처 유지)
                                     │
                          NOTION_API_KEY 있음? ──아니오──▶ lib/data/* 정적 실데이터
                                     │ 예
                                     ▼
                               lib/notion.ts
                     (DB별 fetch · 매핑 · 폴백 · revalidate 3600)
                                     │ 실패 시
                                     └──────────▶ 정적 실데이터로 폴백
```

- `lib/content.ts`의 getter 시그니처는 유지 (CLAUDE.md 계약). 내부만 Notion 분기 추가.
- **소식·갤러리는 병합(union)**: 기존 아카이브(뉴스 47건, 앨범 44개 — 사진 포함)는
  코드에 남기고, Notion에는 새 글부터 작성. 같은 날짜·제목이면 Notion 우선.
  → 사진 500여 장을 Notion에 재업로드하는 비용/리스크 제거. 레거시는 이미 잘 보임.
- 멤버·논문·연구분야·연구과제·모집공고·설정은 **전량 Notion에 시드** — 이후 Notion이
  단일 소스가 되고, 정적 데이터는 장애 폴백으로만 남음.
- Notion 파일 URL은 약 1시간 만료 → `revalidate: 3600`과 맞물려 매 재생성 시 갱신.
  `next.config` `images.remotePatterns`에 Notion S3 도메인 추가.
- 환경변수: `NOTION_API_KEY`, DB ID 8개 — `.env.local` + `.env.local.example` 갱신.

## 10. 교수님용 사용 가이드 목차 (Notion 페이지로 제작, 스크린샷 포함)

1. 5분 훑어보기 — TUPA CMS가 뭐고 어디를 고치면 되는지
2. 새 학생이 왔어요 — 멤버 추가하기 (사진 넣는 법 포함)
3. 학생이 졸업했어요 — 구분을 '졸업생'으로 바꾸고 진로 적기
4. 논문이 나왔어요 — 논문 한 줄 추가 (TRB 시즌 체크리스트)
5. 소식 올리기 — 행을 열어 글 쓰고 사진 붙여넣기
6. 갤러리 앨범 만들기
7. 모집공고 열고 닫기
8. 사이트 문구·이메일·주소 바꾸기 (사이트 설정)
9. 왜 바로 반영이 안 되죠? — 최대 1시간 (급하면 새로고침 URL 안내)
10. 실수해도 괜찮아요 — 비워두면 사이트가 알아서 숨기거나 기본값을 씁니다
11. 뭔가 이상할 때 체크리스트 (공개 체크했나요? → 1시간 기다렸나요? → 연락처)

## 11. 생성 전 필요한 것 (사용자 준비)

- [ ] notion.so/my-integrations 에서 Internal Integration 생성 → **API 키**
- [ ] Notion에 **"TUPA CMS" 부모 페이지** 생성 → ⋯ → Connections 에서 integration 연결
- [ ] API 키를 `.env.local`의 `NOTION_API_KEY`로 전달 (프론트 fetch용 — DB 생성 자체는
      이미 연결된 Notion MCP로 Claude가 직접 수행 가능)
